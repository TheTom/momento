Honest answers. No hedging.

Compatibility
Does Momento work with ChatGPT or Claude in the browser?
No.

Momento runs locally on your machine and integrates with AI tools that support local tool access, such as Claude Code or Codex Desktop.

Web-based chat interfaces like chat.openai.com or claude.ai do not allow third-party local tools to register with the model. Without that capability, Momento cannot connect to them. Momento requires an environment where the AI agent can call local tools directly.

Could Momento ever work with web interfaces?

If web-based AI tools add support for local tool registration, Momento could integrate. They do not today.

A browser extension that injects context is technically possible but fragile, dependent on DOM structure, and breaks whenever the provider changes their UI. That is not something we would ship.

Data & Privacy
Why is Momento local-only?

Momento is designed to keep your project memory on your machine. No cloud sync, no external storage, no third-party access. Your project context stays under your control.

Your data never leaves your device. That is a design choice, not a limitation.

Where is data stored?

A single SQLite file at ~/.momento/knowledge.db. One file. Zero dependencies. You can inspect it, back it up, or delete it at any time.

Does Momento send data to any server?
No.

Momento is a local MCP server. It reads and writes to a local SQLite file. It makes no network requests. It has no telemetry, no analytics, no phone-home behavior.

How It Works
What does Momento actually store?

Structured memory, not chat history. Momento stores five types of entries:

session_state — Where you left off. Current task, recent progress, next step.

decision — What was chosen, why, what was rejected, and the implications.

plan — Finalized plans with phasing and rationale.

gotcha — Bugs, pitfalls, and non-obvious constraints you've already hit.

pattern — Recurring approaches or conventions in your project.

Each entry is concise and actionable. Momento does not replay conversations. It restores the distilled state of your project. Restore is deterministic: the same project state produces the same retrieval ordering every time.

How does the agent know when to save?

Cadence-based checkpointing. The agent is instructed to save after completing significant work: a file change, a resolved bug, a finalized decision. These are observable events the model reliably recognizes.

Momento does not depend on context window introspection or token thresholds. Frequent small saves are more reliable than threshold-triggered panic saves.

How does restore know what to return?

Restore mode follows a fixed priority order. Session state first, then plans, decisions, gotchas, patterns, and finally cross-project entries. This ordering is deterministic, not search-ranked. Within each tier, entries are sorted by surface match, then branch match, then recency.

In monorepos, Momento detects the current working directory and prefers entries matching that surface. If you're in /server, backend checkpoints surface first. If you're in /ios, iOS checkpoints lead. This is a preference, not a filter.

Does it work across different agents on the same project?
Yes.

Momento is agent-agnostic at the data layer. If Claude Code saves a checkpoint in the morning and Codex retrieves context in the afternoon, the restore works identically. The memory layer stays constant. The reasoning engine can change.

Does Momento isolate branches?
No.

Branches are not isolated. Momento captures the current branch at save time and prefers entries from the current branch at restore time, but shared decisions and patterns remain visible across all branches. Memory is ranked, not partitioned.

This means a gotcha saved on main still appears when you're on a feature branch — it's just ranked lower than branch-specific entries. Universal knowledge stays accessible.

Scope
Is Momento only for coding?
No.

Momento stores decisions, reasoning, and task state. That applies to any project that spans multiple sessions: writing, research, strategic planning, product launches, academic work. If it's too complex for one sitting and you care about continuity, Momento helps.

Is Momento a "second brain" or personal knowledge base?
No.

Momento is project memory for AI workflows. It captures knowledge from working sessions so that context survives across sessions and tools. It is not a replacement for Obsidian, Notion, or a notes app.

Does Momento decide what to remember?
No.

The agent decides when to call. Momento decides what to return. There is no autonomous learning or background analysis. Every entry is either created by an explicit agent action or by the developer via CLI.

Agents decide when to call. Momento decides what to return. Clean separation. No magic.

Can I see and manage what's stored?
Yes.

The CLI gives you full visibility: momento inspect to browse entries, momento prune to clean up, momento log to add entries manually. The SQLite database is a standard file you can query directly if you prefer.

What happens if I delete Momento?

Nothing breaks. You lose saved memory, but your AI tools continue working normally. Momento augments sessions; it does not replace them.

Does Momento reduce token usage?

Indirectly, yes. Without persistent memory, every new session starts with you re-explaining architecture, restating decisions, and reconstructing what happened last time. That repeated context burns tokens — often 5,000-8,000 per re-orientation.

Momento restores concise, structured project state in under 2,000 tokens. No multi-turn clarification loops. No redundant context loading. Over weeks of daily use, the savings compound. But the primary benefit is continuity and clarity. Lower token usage is a side effect, not the goal.

What happens if two sessions save at the same time?

Momento stores entries with timestamps and surface tags. Restore always prefers the most recent relevant session state, with surface-matching entries ranked first. There is no conflict resolution needed because Momento is additive, not transactional.

Why build this? Won't AI providers solve memory eventually?

Eventually, probably. Today, they don't. And when they do, it will likely be cloud-based, vendor-locked, and opaque. Momento exists because project continuity matters now, and it should be local, portable, and under your control.

The Cynical Questions
Can't I just use the agent's chat history?

You can. Until it disappears.

Chat history is agent-specific, UI-bound, and often lost on /clear, compaction failure, or session reset. It is not structured for reuse across tools.

Momento stores distilled, structured knowledge in a local, portable database. It survives /clear. It survives tool switching. It survives restarts.

Chat history is a transcript. Momento is durable intent.

Can't I just use git commit messages?

Commit messages explain what changed. They rarely explain why it was chosen, what was rejected, known pitfalls, or in-progress work before commit.

Momento captures decisions and task state before code is finalized. It complements git. It does not replace it.

Why not just add logging to my app?

Application logging captures runtime behavior. Momento captures reasoning about architecture, constraints, and patterns.

Logs answer: "What happened?"

Momento answers: "Why did we build it this way?"

Different layer. Different purpose.

Why not just store everything in a JSON file?

You could. You would also lose full-text search, indexed retrieval, deterministic restore ordering, and concurrency safety. Then you would reimplement a database badly.

SQLite is a single file. It is simpler than inventing your own persistence format.

Why not use a vector database?

Because the core problem is not semantic search. The core problem is restoring working state after context loss.

BM25 + tags + deterministic tier ordering is faster, predictable, easier to reason about, and has zero external dependencies.

Vectors are useful when cross-project semantic recall becomes dominant. That is a later optimization, not a v0.1 requirement.

Can't I just write better prompts?

Better prompts do not survive context resets. They do not share knowledge across agents. They do not remember last week's decision.

Prompts shape behavior. Momento preserves memory.

Isn't this just note-taking?
No.

Notes are written for humans. Momento entries are written for agents: structured, short, tagged, ranked, and bounded by token budget. The output is optimized for machine rehydration, not human reading comfort.

Isn't this overkill for a solo developer?

If you never switch agents, lose context, work across multiple branches, forget why you made a decision, or re-solve the same bug twice — then yes, you don't need it.

If any of those happen weekly, it pays for itself quickly.

What happens if the agent ignores the instructions?

Nothing breaks. The memory layer works independently. You can run momento save, momento status, and momento retrieve from the CLI at any time.

Adapter automation improves cadence. It is not required for correctness.

Does this store my code?
No.

Momento stores decisions, plans, gotchas, patterns, and task checkpoints. It does not index or store your source code. It stores reasoning about your code.

Will this become another knowledge graveyard?

Only if you log everything. Momento is built around hard size limits, explicit logging, confidence weighting, session-state decay, and manual pruning.

The goal is signal density, not accumulation.
